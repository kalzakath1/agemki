
SCUMM ADVENTURE EDITOR

Documento Global del Proyecto

Version v32 — Marzo 2026

Editor de aventuras graficas estilo LucasArts (SCUMM) para MS-DOS.
Desarrollado con Electron + React + Zustand en el editor, y motor C
compilado con Open Watcom 2.0 para DOS en modo 13h (320x200, 256 colores).

1. Stack Tecnologico
1.1 Editor (Windows)
    • Electron + Vite + React 18 + Zustand
    • Node.js para generacion de DATs y main.c
    • Open Watcom 2.0 (wcc386 + wlink) para compilar el motor C
    • DOSBox-X para ejecutar el juego compilado

1.2 Motor C (MS-DOS)
    • Modo 13h: 320x200, 256 colores
    • DOS/4GW (Phar Lap) como extensor DPMI
    • Compilador: Open Watcom 2.0 (wcc386 -bt=dos)
    • Audio: driver MPU-401 propio (mididrv) via puerto 0x330
    • Target: MS-DOS 5.0+

1.3 Audio (nuevo desde v32)
    • Driver MPU-401 propio: mididrv.c / mpu.c / midi.c / timer.c
    • Sustituye completamente a AIL/32 (eliminado)
    • Formato de musica: MIDI estandar (.mid) guardado directamente en AUDIO.DAT
    • Sequenciador MIDI interno con soporte Format 0 y Format 1
    • Timer hook a 1000Hz con chain al ISR del engine (mantiene g_ticks_ms)
    • Cola MPU-401 no bloqueante de 256 bytes, flush de max 32 bytes/frame

3. Motor C — Sistemas Implementados
3.1 Render y modos de pantalla
    • Modo 13h VGA: 320x200 pixels, 256 colores, paleta maestra por juego
    • Fondos PCX cargados desde GRAPHICS.DAT, scroll horizontal de rooms anchas
    • Render de personajes con sprites PCX animados (idle, walk right/left/up/down)
    • Flip horizontal de sprites para economizar assets
    • Escalado de personajes por perspectiva: ScaleZone con interpolacion lineal por Y
    • Render de objetos en escena con estado (visible/invisible)
    • Inventario visual con iconos PCX en barra inferior

3.2 Sistema de verbos (SCUMM-style)
    • Verbsets: conjuntos de verbos configurables (nombre, posicion, color, flags)
    • Verbos con flag isMovement (caminar a) y approach (acercarse antes de ejecutar)
    • Linea de accion: muestra Verbo + Objeto en tiempo real al mover el cursor
    • Hover sobre objeto de inventario: muestra verbo activo + nombre del objeto
    • g_script_running=1 durante handlers para bloquear input del usuario
    • Pending action: walk al objeto y ejecutar handler al llegar

3.3 Sistema de inventario (nuevo/mejorado en v32)
    • InvSlot tiene copia propia del buffer PCX (owns_buf=1) — no comparte puntero con el objeto de escena
    • Al cambiar de room se libera el buffer del objeto pero el inventario conserva su copia
    • Verbos sobre objeto de inventario: busca handler verb_inv, luego respuesta inv_verb, luego verb como fallback
    • Sin verbo activo: hover muestra verbo de movimiento por defecto + nombre objeto
    • Click sin verbo: ejecuta respuesta con verbo de movimiento por defecto

3.4 Sistema usar_con (nuevo en v32)
    • Modo g_usar_con_mode: activo cuando se selecciona verbo + objeto de inventario con handler usar_con
    • Linea de accion cambia a "Usar [objeto] con" esperando segundo objetivo
    • Segundo objetivo puede ser: otro objeto inventario, objeto de escena, o personaje
    • Cancelacion con click derecho o al pulsar cualquier verbo
    • engine_on_verb_inv(verb, inv_obj, fn): handler verbo sobre objeto de inventario
    • engine_on_usar_con(inv_obj, target, fn): handler usar X con Y (target="" = cualquier objetivo)

3.5 Movimiento y pathfinding
    • Walkmap bitmap 40x25 tiles (1=transitable, 0=bloqueado)
    • BFS sobre walkmap para encontrar camino al destino
    • Si no hay camino: aproximarse lo maximo posible + mensaje sys.cannot_reach
    • Velocidad base configurable por personaje, escala con perspectiva

3.6 Sistema de texto y dialogos
    • engine_show_text_ex(key, color, duration_ms): overlay no bloqueante
    • Array de 4 overlays independientes para globos simultaneos
    • Dialogos: DialogueNode con lines[] (hasta 4 lineas simultaneas)
    • Respuestas de verbo: overlay sobre protagonista, no en linea de accion

3.7 Input de teclado (nuevo en v32)
    • Flechas: mueven el cursor KB_SPEED=4 px/evento via kbhit/getch
    • Tras mover cursor con teclado se sincroniza cursor hardware del raton con INT 33h AX=04h
    • Enter/Espacio: click izquierdo (deteccion de flanco con prev_kb)
    • Ctrl: click derecho via INT 16h AH=02h shift flags bit 2
    • ESC: menu de pausa (siempre activo, no bloqueado durante scripts)
    • Durante g_script_running: flechas, enter/espacio y clicks bloqueados. ESC sigue activo

3.8 Scripts bloqueantes (nuevo en v32)
    • g_script_running=1 durante ejecucion de cualquier handler de script
    • Click izquierdo, click derecho, flechas y enter/espacio ignorados durante script
    • Bucles internos del script (engine_show_text, engine_walk_char) siguen leyendo g_mouse.buttons directamente
    • ESC siempre disponible para el menu de pausa

3.9 Salidas (Exits)
    • Exits como objetos con name_key localizada
    • Hover sobre zona de salida: muestra Verbo Nombre_salida en linea de accion
    • Click verbo movimiento: walk al centro, _check_exits completa el cambio de room

3.10 Sistema de secuencias
    • Secuencias: listas de pasos ejecutados en orden
    • Pasos: change_room, play_midi, show_text, call_sequence, run_dialogue, bgColor...
    • engine_play_rooms(): bucle principal que ejecuta la secuencia inicial y luego el gameloop
    • El audio se lanza desde rooms (engine_play_midi) y se procesa en engine_flip via engine_audio_update

4. Sistema de Audio MPU-401 (nuevo en v32)
AIL/32 eliminado completamente. El nuevo driver mididrv usa MPU-401 directamente sin dependencias externas.

4.1 Arquitectura del driver mididrv
Fichero	Responsabilidad
mididrv.c/h	API publica: install, load_mid, play, stop, pause, state, volume, process
mpu.c/h	MPU-401 UART mode. Cola circular 256 bytes no bloqueante. mpu_flush() envia max 32 bytes/frame
midi.c/h	Parser MIDI Format 0/1. Secuenciador de eventos. mpu_send() por cada evento
timer.c/h	Hook IRQ0 a 1000Hz. Chain al ISR del engine cada 55 ticks. timer_process() avanza el secuenciador
agemki_audio.c/h	Wrapper del engine. engine_audio_update() llama mdrv_process() + mpu_flush()

4.2 Integracion en el engine
    • engine_audio_init(NULL, NULL, vol, sfx_vol): inicializa mdrv_install()
    • engine_play_midi(id): carga MIDI del DAT y llama mdrv_load_mid() + mdrv_play()
    • engine_audio_update(): llamado desde engine_flip() cada frame
    • engine_audio_shutdown(): llamado desde engine_quit(), para audio y restaura timer
    • AUDIO.DAT contiene MIDI estandar (MThd), no XMI
    • DOSBox requiere mpu401=intelligent en dosbox.conf

4.3 Problema resuelto: timer del engine
Nuestro driver reprogramaba el PIT a 1000Hz sobreescribiendo el ISR del engine. Esto causaba que g_ticks_ms dejara de avanzar y los bucles de movimiento se colgaran indefinidamente.

    • Fix: timer.c usa _chain_intr(g_old_isr) cada 55 ticks para que el ISR del engine reciba sus interrupciones a 18.2Hz
    • engine_audio_update() en engine_flip() en lugar de engine_loop() porque los scripts llaman funciones de room que no pasan por engine_loop

5. Editor — Modulos UI
5.1 Modulos principales
    • SceneSidePanel: edicion de rooms (fondo, walkmap, objetos, exits, scale zones, MIDI)
    • CharEditor: personajes con sprites por direccion, velocidad, flag protagonista
    • VerbsetEditor: creacion de verbsets con verbos, colores, layout en grid
    • SequenceEditor: secuencias de pasos con UI drag/drop
    • DialogueEditor: arboles de dialogo con nodos y conexiones
    • ObjectEditor: objetos con tabs General, Respuestas, Inv. respuestas, Combinar (reestructurado en v32)
    • ScriptEditor: scripts con triggers incluyendo verb_inv y usar_con (nuevos en v32)
    • GameParams: paleta maestra, audio, textos del sistema
    • LocaleEditor: textos del juego por idioma

5.2 Modulo Objetos (reestructurado en v32)
    • Tab General: nombre, tipo, detectable, estados visuales con sprites
    • Tab Respuestas: para cada verbo del verbset activo, elige texto (localizable, clave obj.ID.verb.VERB_ID) o script (dropdown de scripts existentes)
    • Tab Inv. respuestas: igual que Respuestas pero clave obj.ID.inv_verb.VERB_ID, solo activo cuando el objeto esta en inventario. Fallback a Respuestas si no hay respuesta de inventario
    • Tab Combinar: lista de entradas con objetivo (objetos + NPCs) y script a ejecutar. Genera engine_on_usar_con en el build
    • Eliminados: tabs Scripts y Flags del modulo Objetos (el modulo Scripts independiente queda intacto)

5.3 Nuevos triggers en ScriptEditor
    • verb_inv (Verbo + Inventario): dispara script cuando el jugador aplica un verbo a un objeto del inventario. Genera engine_on_verb_inv()
    • usar_con (Usar inv. con...): dispara script cuando el jugador usa un objeto de inventario con otro objetivo. Campo objetivo opcional (vacio = cualquier objetivo). Genera engine_on_usar_con()

5.4 Sistema de build
    • Build completo: genera DATs, compila C con mididrv (mpu.obj midi.obj timer.obj mididrv.obj), linka, lanza en DOSBox
    • AUDIO.DAT: guarda MIDI directamente sin convertir a XMI (convertMidToXmi eliminada)
    • index.js genera engine_on_verb_object para verbResponses mode=script
    • index.js genera engine_on_verb_inv para invVerbResponses mode=script
    • index.js genera engine_on_usar_con para combinations
    • Log de build en tiempo real en la UI

5.5 Convenciones activas
    • REGLA: si el usuario pone ? -> solo hablar, no escribir codigo
    • base_speed = velocidad base del personaje; speed = velocidad del walk actual
    • Pending se asigna ANTES de engine_walk_char_to_obj
    • Handler manual con fn != NULL tiene prioridad sobre isPickup automatico
    • g_script_running=1 durante ejecucion de handlers
    • Overlay sobre protagonista para respuestas de verbo (no linea de accion)
    • AUDIO.DAT contiene MIDI estandar; se carga con engine_dat_load_audio
